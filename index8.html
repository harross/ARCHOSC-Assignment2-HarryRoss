OS Memory Management

Task 1 asked us to calculate the properties of a virtual address space with a 4-bit VPN and 8-bit offset, giving 16 virtual pages of 256 bytes each.

Task 2 split the bottom 32KB of physical memory into 8 frames of 4KB each, with the following address ranges:

Frame 0: 0x0000 - 0x0FFF
Frame 1: 0x1000 - 0x1FFF
Frame 2: 0x2000 - 0x2FFF
Frame 3: 0x3000 - 0x3FFF
Frame 4: 0x4000 - 0x4FFF
Frame 5: 0x5000 - 0x5FFF
Frame 6: 0x6000 - 0x6FFF
Frame 7: 0x7000 - 0x7FFF

Task 4 worked through TLB sizing: for a 32-bit virtual, 24-bit physical address space with 1KB pages, the page table requires 4M entries at 2 bytes each, occupying 8,192 pages of physical memory.

I then extended the example class to simulate a 4-entry FIFO TLB, tracking hits and misses over a sequence of accesses:

public int? Lookup(int vpn) {
    foreach (var entry in entries)
        if (entry.VPN == vpn) { Hits++; return entry.PPN; }
    Misses++;
    return null;
}

public void Insert(int vpn, int ppn) {
    if (entries.Count >= capacity) entries.RemoveAt(0);
    entries.Add(new TLBEntry { VPN = vpn, PPN = ppn });
}

Running the access sequence 0,1,0,2,3,0,1,4,5,0,1,6,7,0,1 against the 4-entry TLB produced a 33.3% hit ratio, as VPNs 0 and 1 are repeatedly evicted once the working set grows beyond 4 entries:

Hits: 5, Misses: 10
Hit ratio: 33.3%

Today, I covered virtual memory translation, page frames, and TLB behaviour, showing how a small TLB capacity directly limits hit ratio when the working set exceeds it.
